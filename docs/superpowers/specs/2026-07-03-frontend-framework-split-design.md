# Frontend Convention Framework Split: Shared Core + Next.js + Vite

Date: 2026-07-03
Status: approved

## Problem

`~/.claude/CLAUDE-FRONTEND.md` defines the ideal web-client directory structure predicated on Next.js App Router. The front-end audit agents (audit-engineering, audit-ux, audit-design, audit-marketing) reference this file by path, so Vite-based apps have no structure rubric to be graded against. The existing structure also drifts from global rules: it prescribes `src/lib/` and a flat `hooks/`, both banned or superseded by R-305/R-306 (`api/`, `clients/`, `services/`, `state/` vocabulary).

## Decisions

1. Split into three files: a framework-agnostic shared core plus one file per framework.
2. Vite target: React SPA with TanStack Router (file-based routing). Pairs with the existing TanStack Query mandate.
3. Correct both framework structures to R-305 vocabulary during the move. `lib/` is eliminated everywhere.

## Deliverables

All files live in `~/.claude/` (public repo; content is generic, no client-identifying material).

### 1. `CLAUDE-FRONTEND.md` (rewritten as shared core)

Keeps everything framework-agnostic:

- Stack rules: React 19, TypeScript strict, SCSS modules, TanStack Query, no Tailwind
- Component patterns (file structure, props interfaces, exports, no React.FC)
- Import ordering (generalized: framework imports replace the Next.js-specific group)
- File naming table (framework-specific rows move out)
- TypeScript patterns
- State management
- API-wrapper contract (base URL sourced per framework file; behavior rules stay here)
- Error handling
- Prettier config and ESLint rules
- E2E test layout (Playwright, user-story-driven)

Gains a `## Framework Files` section at the top:

- `next` in `package.json` dependencies: read `CLAUDE-FRONTEND-NEXT.md`
- `vite.config.ts` in project root: read `CLAUDE-FRONTEND-VITE.md`

### 2. `CLAUDE-FRONTEND-NEXT.md` (new)

Framework-specific content moved from the current file, corrected to R-305:

- App Router directory structure with route groups `(auth)`, `(protected)`
- Corrections: `lib/api.ts` becomes `api/` modules (one fetch wrapper per backend route), `lib/queryClient.ts` becomes `config/queryClient.ts`, `hooks/` folds into `state/`, `lib/` removed
- `page.tsx` / `layout.tsx` / `loading.tsx` / `error.tsx` naming
- Metadata export and `next/font/google` patterns
- `'use client'` directive rules
- `NEXT_PUBLIC_*` env vars

### 3. `CLAUDE-FRONTEND-VITE.md` (new)

React SPA + TanStack Router, R-305-compliant from birth:

```
src/
├── main.tsx                  # Entry: createRoot + RouterProvider
├── routeTree.gen.ts          # Generated by TanStack Router; never hand-edited
├── routes/                   # File-based routes
│   ├── __root.tsx            # Root route: providers, layout shell
│   ├── index.tsx
│   ├── login.tsx
│   └── _protected/           # Pathless layout route for authed pages
│       └── dashboard.tsx
├── components/<PascalCase>/  # ComponentName.tsx + ComponentName.module.scss
├── features/<name>/          # Feature slices
├── api/                      # Own-backend fetch wrappers, one per route
├── clients/                  # Third-party SDK wrappers
├── services/                 # Domain logic
├── state/                    # Stores, hooks, context providers; no separate hooks/
├── config/                   # queryClient.ts, router config, env parsing
├── constants/
├── data/
├── styles/                   # globals.scss, design tokens
└── types/
```

Additional rules:

- Directories appear only when occupied (R-309); the vocabulary is fixed, not mandatory on day one
- Route URL segments keep kebab-case per the R-312 exception; all other directories camelCase
- Entry chain: `index.html` at project root, `src/main.tsx` mounts the router
- Env vars: `VITE_*` prefix, accessed via `import.meta.env`, parsed/validated in `config/`
- `@/` path alias configured in `vite.config.ts` and `tsconfig.json`
- No `'use client'`; every module is client-side
- Base URL for the API wrapper: `VITE_API_URL`

### 4. Reference updates

- `~/.claude/CLAUDE.md` convention-file table: add rows for `CLAUDE-FRONTEND-NEXT.md` and `CLAUDE-FRONTEND-VITE.md`
- `~/.claude/rules/session-types.md` stack-detection table: note the framework files next to the frontend entry
- `~/.claude/agents/audit-engineering.md` convention list: add both framework files
- Other audit agents reach the framework files through the core file's `## Framework Files` pointer; no edits

## Out of scope

- Migrating existing Next.js projects off `lib/` and `hooks/`. The next engineering audit on those projects will flag structure findings; that is the intended effect. Migration is separate work.
- Non-React Vite targets (extensions, vanilla TS).
- Enforcement hooks (R-516): these are convention docs read by agents, not mechanizable per-edit rules.

## Verification

## Delivery

This spec commits on its own. Implementation lands as one commit to `~/.claude` containing all three convention files plus reference updates. No push until asked (batch-deploy preference).
